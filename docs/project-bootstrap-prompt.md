# 프로젝트 부트스트랩 프롬프트

이 프로젝트(예배용 악보 관리 + 콘티 구성 앱)를 **처음부터 다시 만든다면** 쓸 프롬프트.
수개월간 실사용 피드백으로 얻은 결정사항을 처음부터 명세에 박아넣어, 같은 시행착오를
반복하지 않게 하는 것이 목적이다. (원 자료: `CLAUDE.md` + `docs/worklog/`의 "팁" 섹션)

> 사용법: 아래 "프롬프트" 블록을 그대로 붙여넣고, 진행 방식 섹션대로 단계별 검증을 시키면
> 골격은 한 세션에 잘 나온다. "한 방에 전부"보다 **단계별 구동 검증**이 결과물이 훨씬 좋다.

## 프롬프트

```
예배용 악보 관리 + 콘티(셋리스트) 구성 웹앱을 만들어줘.

## 기술 스택 (고정)
- 백엔드: Java 17 + Spring Boot 3.x + Gradle, Flyway 마이그레이션
- 프론트엔드: Vue 3 + TypeScript + Vite + Tailwind v4 (shadcn-vue 스타일)
- DB: PostgreSQL / 스토리지: Cloudflare R2 (STORAGE_TYPE=local|r2 전환 가능하게)
- 배포: 백엔드 Fly.io(GitHub Actions), 프론트 Vercel(Git 연동), DB Supabase

## 도메인 모델
songs(곡) → song_sheets(악보 버전, 키별, sort_no로 순서) → song_files(파일)
setlists(콘티) → setlist_items(곡 순서 + 어떤 악보 버전을 쓸지 + 연주키 메모)
- 모든 삭제는 soft delete(deleted_at)
- 파일은 UUID로 저장, 원본 파일명 별도 보관
- 같은 곡 안에서 동일 키 중복 허용

## 핵심 기능
1. 곡/악보/파일 CRUD + 통합 검색(제목·아티스트·가사 LIKE, 키 필터, 태그 필터)
2. 곡+악보 1단계 통합 등록, 일괄 업로드(드래그앤드롭 + 카드별 편집)
3. 콘티: 드래그앤드롭 순서 변경, PDF 다운로드(jsPDF), 진행 모드(전체화면 슬라이드)
4. 공유: UUID 토큰 기반 공개 링크(/share/:token), 비로그인 열람
5. 가사(직접 편집), 태그 칩, 멀티 플랫폼 링크(YouTube 자동 감지+임베드)
6. 악보 뷰어: 이미지 슬라이드, 키보드 ←/→, 터치 스와이프, 스타일러스 필기(벡터 저장)
7. 인증: JWT 자체 구현(30일), 게스트 로그인, 관리자 롤(사용자 관리+휴지통+복구)
   - 공개 API는 auth/**, 공유 뷰, 파일 view/download만. 나머지 전부 인증.

## 처음부터 지켜야 할 것 (시행착오로 배운 것들)
- 투명 배경 PNG는 다크 테마 뷰어에서 검게 보인다 → 업로드 시 흰 배경으로
  평탄화(WebP q0.95, 미지원 브라우저는 PNG 폴백). 악보엔 JPEG 쓰지 말 것(선 번짐+용량 손해).
- jsPDF는 WebP를 못 읽는다 → PDF 생성 시 canvas로 JPEG 재인코딩 거칠 것.
- 모바일(375px) 필수 검증: 헤더/버튼 라벨에 whitespace-nowrap, 좁은 화면에선
  라벨을 아이콘화(hidden sm:inline + aria-label). 가로 스크롤 0 확인.
- DB 커넥션 풀: 서버리스/suspend 환경 대비 HikariCP keepalive·max-lifetime을
  풀러(Supabase pooler) idle timeout보다 짧게.
- 에러 피드백은 toast로 통일(성공·실패 모두), catch{}로 삼키지 말 것.
- 다크 기본 테마 + CSS 변수 토큰 시스템, FOUC 방지(index.html 인라인 스크립트로 dark 선적용).
- UI 텍스트는 한국어, 존댓말 해요체 통일.

## 진행 방식
단계별로 나눠 진행하고(스키마 → 백엔드 API → 프론트 화면 → 인증 → 배포),
각 단계마다 실제로 구동해 검증한 뒤 다음으로 넘어가줘.
```

## 왜 이렇게 쓰는가 (배경)

- **"악보 앱 만들어줘"로는 부족하다.** 이 프로젝트의 진짜 가치는 마이그레이션 V1~V18이
  보여주듯 *실사용 피드백으로 다듬은 부분*(콜드스타트 대응, 투명 PNG 문제, 모바일 깨짐,
  DB 커넥션 풀 튜닝 등)이다. 그 결정들을 "처음부터 지켜야 할 것"으로 명세에 넣는 게 핵심.
- **"한 번에 전부"의 함정.** 위 프롬프트로 골격은 한 세션에 나오지만, "진행 방식" 섹션으로
  단계별 구동 검증을 강제하는 편이 한 방에 뽑는 것보다 결과가 훨씬 안정적이다.
- **재사용.** 비슷한 프로젝트를 새로 시작할 때는 그 도메인의 CLAUDE.md + worklog "팁"을
  같은 방식으로 긁어 "처음부터 지켜야 할 것"에 넣으면 시행착오 비용을 안 내고 시작할 수 있다.

## 이 저장소에서 실제로 겪은 함정 모음 (worklog 팁 발췌)

- 투명 PNG vs 흰배경 평탄화본은 **흰 배경 위에서 원리적으로 구분 불가**(픽셀이 같아짐).
  판별하려면 합성된 렌더가 아니라 알파 채널이 살아있는 원본을 디코드해야 한다.
- "어둡다/투명하다"를 colorType으로 판단하지 말 것. RGBA여도 알파가 전부 255면 불투명,
  불투명이어도 배경이 검정이면 어둡다 → **투명 스캔과 밝기 스캔은 별개 문제**.
- 파일 삭제/교체 전 **콘티(setlist_items) 참조부터 확인**. 겉보기 정상 세트가 아니라
  콘티가 실제 참조하는 세트를 고쳐야 화면이 바뀐다.
- Cloudflare Quick Tunnel URL은 컨테이너 재시작마다 바뀐다 → 프론트가 죽은 백엔드를
  가리키는 반복 장애의 원인. 고정 주소(Fly.io 등)로 두는 게 근본 해결.
- 대량 프로덕션 덮어쓰기는 **드라이런 → 1건 테스트 → 전량**, 재실행 가능(idempotent)하게.
- dev 서버(Vite HMR)는 특정 요소만 stale 렌더로 남길 수 있다 → "버그처럼 보이는 것"은
  동일 조건 대조 테스트로 오탐을 거를 것. 프로덕션 빌드가 진짜 기준.
