# 2026-07-04 작업일지

## 결정한일

- **곡 병합 UI(#171) 방식**: 백엔드 병합 API(`POST /api/songs/{targetId}/merge`, `SongMergeService`)는 이미 있었으므로 새로 만들지 않고, 곡 목록에 "선택 모드"(체크박스)를 추가해 여러 곡을 고른 뒤 "남길 대상 곡"만 선택하면 나머지를 병합하는 화면을 얹기로 함.
- **악보 파일 표시 파일명(#161) 방식**: 카메라가 붙이는 원본 파일명(`IMG_1234.jpg`)을 그대로 노출하지 않고, 업로드 시점에 **`곡제목_키.확장자`** 형태로 새로 생성해 저장하기로 함. 원본 스토리지 키(`stored_file_name`, UUID)는 그대로 두고 화면 표시용 이름만 바꾸는 방식이라 스토리지 구조에는 영향 없음.
- **악보 정보 수정 시 파일명 미갱신 버그(#174) 대응 범위**: 사용자가 스크린샷으로 "키·버전을 바꿨는데 파일명은 그대로"라고 제보 → 원인(파일명은 업로드 시점에만 한 번 생성되고 이후 정보 변경에는 반응하지 않음)을 파악한 뒤, 사용자에게 "화면 표시 파일명 자체를 바꿀지" 확인하고, 형식은 **`곡제목_키_버전`**(추천안)으로 확정. `stored_file_name`·스토리지 키는 유지하고 표시 파일명만 재생성하는 방향으로 결정.
- **악보 버전 리스트 UX(#176) 방식**: 사용자가 직접 제안한 3가지(①버전 클릭 시 해당 상세 포커싱 ②하나 펼치면 나머지는 자동으로 접힘 ③드래그로 순서 변경 + 자동 저장)를 그대로 채택. 순서 변경을 서버에 영구 저장하기 위해 `song_sheets`에 `sort_no` 컬럼을 새로 추가하기로 함(V18).
- **콘티→곡 상세 이동 후 복귀 버그(#177) 해결 방식**: 별도 상태 저장소 없이, 이동 시 쿼리스트링(`query.from`)에 출처 경로를 실어 보내고 곡 상세에서 그 값으로 돌아가는 방식을 선택. 단, 오픈 리다이렉트 취약점을 막기 위해 **내부 경로(`/`로 시작, `//`로 시작하지 않음)만** 허용하도록 검증 추가.
- **악보 관리 "보기" 버튼(#178) 처리**: 이미 상단 슬라이드 뷰어에서 이미지가 보이는 상태라 목록의 "보기" 버튼은 중복이고, 오히려 파일 타입에 따라 사실상 다운로드처럼 동작해 혼란을 준다는 사용자 피드백을 받아 — 별도 대안을 만들지 않고 **버튼 자체를 제거**하는 쪽으로 결정.

## 세부내역

### #171 → #172 — 곡 목록에서 중복 곡 합치기(병합) UI
- `songApi.ts`에 `mergeSongs()` + `SongMergeResponse` 타입 추가.
- `SongList.vue`에 선택 가능 모드(체크박스) 지원 — 선택 모드에선 카드가 상세 이동 대신 선택 토글로 동작(`RouterLink`↔`button` 동적 전환).
- `SongListView.vue`: 헤더에 "곡 합치기" 선택 모드 토글, 하단 고정 액션 바(선택 개수/"합치기(N)", 2곡 이상 선택 시 활성), 남길 대상 곡을 라디오로 고르는 모달 → 병합 실행 → 목록 새로고침 + 태그 갱신 + 결과 토스트.
- 백엔드 병합 자체는 기존 로직 재사용: 대상 곡으로 나머지 곡의 악보·콘티 참조를 이관하고 원본 곡은 soft delete, `dedupeSheets=true`로 키+파일 동일 악보 중복도 정리.
- 로컬 e2e(Playwright): 3곡 중 2곡 선택 → 병합 → 목록이 2곡으로 갱신, 대상 곡만 남고 성공 토스트 노출 확인.

### #161 → #173 — 악보 파일명 자동 생성
- `SongFileService.uploadFile`에서 표시용 `original_file_name`을 `곡제목_키.확장자`로 새로 생성. 키가 없으면 `곡제목.확장자`, 같은 악보에 파일이 여러 개면 순번(`_2`, `_3`) 부여.
- 파일명 부적합 문자(경로 구분자·제어문자) 제거, 앞뒤 공백/점 정리. 확장자는 업로드 원본명의 마지막 경로 세그먼트에서만 추출(슬래시 혼입 방지).
- `stored_file_name`(UUID)·실제 저장 경로는 그대로 유지 — 스토리지엔 영향 없음.
- 로컬 e2e: "은혜의 강가로"+키 G → `은혜의 강가로_G.png`, 같은 악보 두 번째 파일 → `은혜의 강가로_G_2.png`, 키 없는 곡 → `제목만.png` 확인. 백엔드 테스트 40건 통과.

### #174 → #175 — 악보 정보 수정 시 파일명 자동 재생성
- 자동 파일명 형식을 `곡제목_키_버전`으로 확장(키·버전 없으면 해당 부분 생략).
- `SongFileService.regenerateFileNamesForSheet(sheet)`: 악보의 활성 파일들을 등록 순서(songFileId) 기준으로 현재 정보에 맞춰 파일명 재생성. `stored_file_name`·스토리지 키는 그대로 두고 표시 파일명만 바꿈.
- `SongSheetService.updateSheet`(키/버전 변경 시), `SongService.updateSong`(곡 제목 변경 시 그 곡의 모든 활성 악보 파일 재생성)에서 각각 호출.
- `SongFile.rename()` 메서드 추가.
- 로컬 e2e: 업로드 `실로암_A.png`/`실로암_A_2.png` → 키 G·버전 "마커스"로 수정 → `실로암_G_마커스.png`/`실로암_G_마커스_2.png` → 곡 제목 변경 시에도 자동 갱신 확인. 백엔드 테스트 40건 통과.

### #176 — 악보 버전 리스트 아코디언 + 드래그 순서변경
- 백엔드: `V18` 마이그레이션으로 `song_sheets.sort_no` 컬럼 추가(기존 순서 백필), `SongSheet.sortNo` + `Song.sheets @OrderBy(sortNo, songSheetId)`, 신규 버전은 맨 뒤 배치. `PATCH /api/songs/{songId}/sheets/reorder`로 순서 목록을 받아 `sort_no` 재부여. 버전 목록/병합 조회 정렬을 `sortNo` 기준으로 변경.
- 프론트엔드: `reorderSongSheets` API 추가. `SongDetailView`의 "악보 관리" 버전 카드를 아코디언으로 전환(헤더 클릭 시 펼침/접힘, 한 번에 하나만 열림), 손잡이(⋮⋮) 드래그로 순서 변경 시 즉시 자동 저장. 드래그는 Pointer Events 기반이라 마우스/터치/펜 공통 동작.
- 로컬 e2e: A,B,C 버전 생성 → `[C,A,B]`로 순서 변경 → 조회 순서 `C,A,B` 확인. 아코디언/드래그의 실제 조작감은 Vercel 프리뷰에서 사용자가 최종 확인.

### #177 — 콘티에서 연 곡 상세의 "목록으로" 복귀 버그
- `SetlistDetailView`에서 곡 상세로 이동할 때 `query.from`에 출처 콘티 경로(`/setlists/{id}`)를 실어 보냄.
- `SongDetailView`는 `query.from`이 있고 내부 경로(`/`로 시작, `//`로 시작하지 않음)일 때만 그 경로로 돌아가는 버튼("돌아가기")을 보여주고, 없으면 기존처럼 "목록으로"(`/songs`).
- 곡 목록·검색 등 다른 경로로 진입한 경우는 `from`이 없으므로 기존 동작 그대로 유지.

### #178 — 악보 관리 파일 목록 "보기" 버튼 제거
- 파일 목록 행에서 "보기"(Eye) 버튼만 제거, 다운로드·삭제는 유지.
- `Eye` 아이콘 자체는 상단 전체화면 뷰어 버튼에서 계속 쓰이므로 import는 유지.

## 팁(알아두면 좋은 정보)

- **여러 기능을 연달아 작은 PR로 쪼개 빠르게 병합하는 흐름이 이번에 잘 맞았음** — 이슈 하나당 커밋 1개·PR 1개로 끝나는 규모로 유지하니 병합까지 몇 분이면 끝났고, 사용자가 스크린샷으로 문제를 제보하면 바로 다음 PR로 이어붙이기 수월했음.
- **파일 표시명과 저장 파일명(스토리지 키)을 분리해둔 설계 덕분에**, "표시명을 곡 정보에 맞게 자동 재생성"하는 기능(#161, #174)을 나중에 추가로 확장하기가 쉬웠음 — 스토리지 쪽은 전혀 건드릴 필요가 없었음.
- **`query.from` 같은 "돌아갈 경로"를 쿼리스트링으로 넘기는 패턴은 오픈 리다이렉트 취약점을 유발하기 쉬움** — 반드시 "내부 경로인지"(`/`로 시작하고 `//`로 시작하지 않는지)를 검증한 뒤에만 리다이렉트에 사용할 것.
- **사용자가 UX 아이디어를 구체적으로 3단계로 적어주면(예: #176), 별도 설계 고민 없이 그대로 구현 스펙으로 옮길 수 있어 훨씬 빠름.**
