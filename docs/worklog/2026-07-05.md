- **"PNG라서 가끔 어둡게 보이는거 아니냐"는 사용자 질문에 대한 진단**: 앱이 다크 테마 기본이라, 알파 채널(투명 배경)이 있는 PNG는 뷰어의 어두운 배경이 비쳐 어둡게 보인다고 진단. 사용자에게 "①뷰어 쪽에 흰 배경 깔기" vs "②업로드/보정 시 알파를 흰색으로 정규화(#179 파이프라인 재사용)" 두 옵션을 제시 → 사용자가 "2"를 선택했고, 이어서 "원래 밝기조정 기능이 2번처럼 하려던 거야"라고 정정 — 즉 별도 파이프라인을 새로 만들지 않고, 이미 만든 밝기 보정의 **저장 로직 자체가** 알파를 평탄화하도록 고치는 것으로 범위를 확정.
- **곡 상세 화면에 밝기 보정이 없던 문제**: 사용자가 스크린샷으로 "곡 상세에서는 안 바뀐다"고 제보 → 원인이 `SongDetailView.vue`의 자체 슬라이드 뷰어가 `SheetViewerModal`을 아예 쓰지 않고, 확대(⤢) 버튼도 모달이 아니라 새 탭 링크였기 때문임을 확인 → 사용자 동의를 받아 곡 상세에도 진입점을 추가하기로 결정.
- **"자동 보정 눌러도 완전한 흰 배경이 안 된다"는 후속 피드백 대응**: 기존 자동 보정이 이미지 **전체 평균 밝기**를 목표값(170)에 맞추는 방식이라, 실제로 어둡게 찍힌 사진(투명이 아니라 진짜 어두운 배경)은 평균이 끌어올려져도 배경 자체가 흰색까지 도달하지 못하는 구조적 한계가 있음을 파악 → **평균 대신 "배경"(밝기순 정렬 시 상위 10% 지점, 악보 특성상 대부분 흰 배경이므로 이 값을 배경으로 간주) 기준으로 목표(250)에 맞추는 방식**으로 알고리즘을 바꾸기로 결정. 동시에 "악보 상세(악보 관리) 화면엔 자동보정이 없다"는 지적을 받아, 악보 관리 파일 목록에도 진입점을 추가하기로 함.

## 팁(알아두면 좋은 정보)

- **컨테이너가 재시작되면 background 로그 watcher·cron 체크인 예약이 전부 사라진다.** 재시작 직후엔 항상 `git status`/`git log origin/main`으로 실제 커밋·PR 상태부터 다시 확인하고, 로컬 `main` 브랜치가 `origin/main`보다 뒤처져 있으면(`git pull origin main --ff-only`) 반드시 최신화한 뒤 새 작업 브랜치를 따야 한다 — 뒤처진 로컬 `main`에서 브랜치를 따면 이미 병합된 기능이 통째로 빠진 코드로 작업하게 된다.
- **PR 머지 후 다음 작업은 반드시 `origin/main`에서 새 브랜치를 따야 한다.** 이번엔 `git checkout -b`를 stale한 로컬 `main` 기준으로 시도했다가 밝기 보정 기능 전체가 빠진 파일을 보고 잠깐 혼동한 적이 있었음.
- **CSS `filter: brightness()`는 곱연산(원본×배율) 방식**이라, 목표를 "평균 밝기"로 잡으면 어두운 잉크가 평균을 끌어내려 배경이 완전한 흰색에 도달하지 못한다. 배경이 대부분인 이미지는 **밝기 상위 퍼센타일(배경으로 추정되는 값)을 기준으로 화이트포인트를 맞추는 방식**이 훨씬 의도에 맞는다.
- **Vercel PR 프리뷰 URL(`*.vercel.app`, PR별로 매번 새로 생김)은 백엔드 CORS 허용 목록에 없어서 로그인 등 API 호출이 항상 실패한다.** 사용자가 프리뷰 링크로 동작 테스트를 하려고 하면, 화면 미리보기 용도일 뿐 실제 로그인/API 테스트는 프로덕션 주소(`worship-sheet.vercel.app`)에서 해야 한다고 안내할 것.
- **Playwright e2e에서 알파/밝기 문제를 검증할 때는 실제 파일 대신 canvas로 합성한 테스트 PNG(투명 절반+불투명 어두운 절반, 또는 배경/잉크 두 톤)를 만들어 저장 전후 픽셀값(RGBA)을 직접 비교**하면, 화면 스크린샷만으로는 확인하기 어려운 "완전 불투명으로 평탄화됐는지", "배경이 정확히 흰색(255)이 됐는지" 같은 것을 정량적으로 검증할 수 있다.
