- **"Fly.io 콜드스타트가 답답하니 회사 데스크탑을 백엔드 서버로 쓰고 싶다"는 요청**: 처음엔 공인 IP 노출까지 고려했으나 "로컬host로 해도 되냐"로 축소, 최종적으로 **Docker 컨테이너 + Cloudflare Tunnel + Vercel 프론트엔드에서 데스크탑 주소로 전환**하는 방향으로 결정. Fly.io는 삭제하지 않고 안전망(폴백)으로 유지하기로 함.
- **스토리지(R2 vs local) 마이그레이션 필요 여부를 잘못 판단했다가 정정**: Fly.io secrets에 `STORAGE_TYPE`/`R2_*`가 전부 설정되어 있는 걸 보고 "운영이 이미 R2일 것"이라 추정, 파일을 프로덕션 API로 하나씩 내려받는 마이그레이션 스크립트를 만들어 실행했다가 사용자가 "뭔가 잘못된 거 같은데"라고 제동을 걺 → 목표(데스크탑도 R2를 그대로 쓰면 파일 이동 자체가 불필요)를 재확인하고 사용자가 "파일 백업 빼고 진행"을 지시, 다운로드 작업은 중단. 이후 데스크탑 컨테이너에 실제 R2 자격증명을 넣어 연결하는 것으로 마무리(스토리지 이전 불필요함을 최종 확인).
- **R2 연결 실패를 단계적으로 진단**: ①TLS handshake_failure(계정 ID 31자, 32자여야 함 — 오타로 확인) → ②`NoSuchBucketException`(버킷 이름 오기) 순서로 원인을 좁혀가며 사용자가 `.env.desktop`을 직접 수정하도록 안내, 최종적으로 R2 파일 다운로드까지 프로덕션과 동일한 바이트 수로 정상 확인.

- **프론트엔드에 "1차 데스크탑 실패 시 2차 Fly.io로 자동 폴백" 로직 도입 결정**: 사용자가 "로컬 PC가 잠들면 Fly.io로 연결되게 할 수 있냐" 질문 → axios 인터셉터 레벨에서 네트워크 에러(서버 응답 자체가 없는 경우)만 폴백 대상으로 삼는 방식을 제안, 트레이드오프(폴백 시 매 요청 타임아웃 대기, JWT_SECRET이 서로 달라 재로그인 필요)를 설명하고 진행 승인받음.

## 팁(알아두면 좋은 정보)

- **Windows Git Bash에서 `node -e`로 `/tmp/...` 경로를 쓰면 실패한다.** MSYS2 bash 네이티브 툴(curl 등)은 `/tmp` 경로를 실제 윈도우 경로로 자동 변환해주지만, `node.exe` 같은 네이티브 윈도우 바이너리는 이 변환에 참여하지 않고 `/tmp/...`를 현재 드라이브 기준 리터럴 경로(`D:\tmp\...`)로 해석해 `ENOENT`가 난다. 세션 스크래치패드 디렉터리처럼 완전한 절대경로를 항상 사용할 것.
- **PowerShell의 `curl`은 `Invoke-WebRequest` 별칭이라 `-X POST` 같은 실제 curl 플래그를 기대한 대로 처리하지 못할 수 있다.** 사용자가 PowerShell에서 `curl -X POST ...`로 405를 만난 사례가 있었음 — `curl.exe`로 확장자를 명시하면 시스템에 설치된 진짜 curl이 실행된다.
- **PR이 머지돼도 브랜치는 자동으로 안 지워진다** (저장소의 "머지 후 자동 삭제" 설정이 꺼져 있으면 원격에 남고, 로컬은 체크아웃된 브랜치가 있으면 계속 보임). 머지 후 다음 작업 전에 로컬을 정리하려면 다른 브랜치/커밋으로 전환한 뒤 `git branch -d`로 지워야 하며, 원격 삭제는 되돌리기 부담이 있으니 사용자의 명시적 지시 없이는 진행하지 않는 편이 안전하다.
- **Vercel의 "Redeploy"는 "Use existing Build Cache"가 켜져 있으면 실제로 다시 빌드하지 않을 수 있다.** Vite는 `import.meta.env.VITE_*` 값이 바뀌면 반드시 새 파일 해시가 나오므로, 배포된 JS 번들 파일명이 그대로인지 확인하면 재빌드 여부를 빠르게 검증할 수 있다. 단, 이번 사례의 진짜 원인은 캐시가 아니라 **로컬에서 만든 코드 변경을 아예 커밋·푸시하지 않은 것**이었다 — 환경변수만 바꾸고 실제 배포 결과가 안 바뀌면, 캐시보다 먼저 `git status`로 커밋 여부부터 확인할 것.
- **Cloudflare 계정 ID(R2 엔드포인트 서브도메인)는 정확히 32자리 16진수다.** 한 글자라도 짧거나 길면 TLS handshake 단계에서 서버가 즉시 거부(`handshake_failure`)하는데, 이때는 인증서조차 안 나오므로 자격증명(액세스 키)이 잘못됐다고 오인하기 쉽다 — `openssl s_client`나 raw `curl -v`로 애플리케이션 레이어 이전(TLS 레벨)에서 실패하는지부터 확인하면 원인을 빠르게 좁힐 수 있다.
- **Vercel 대시보드에서 환경변수를 "Sensitive"로 저장하면 나중에 평문으로 되돌릴 수 없다** — 다시 평문으로 쓰려면 삭제 후 재생성해야 한다. 민감하지 않은 값(공개 URL 등)은 처음부터 Sensitive를 켜지 않는 게 편하다.
