### #197 — "배경 흰색으로"를 WebP로 + PDF 포맷 무관 처리
- 평탄화 출력을 **WebP(q0.95) + PNG 폴백**으로 변경(결과 Blob 실제 type으로 확장자 결정해 바이트·확장자·contentType 일치). 포맷별 실측(966×1089): 원본 투명 PNG 124KB / 평탄화 PNG 208KB / JPEG q0.92 187KB(선 번짐) / **WebP q0.95 97KB**.
- **부수 발견·수정**: 셋리스트 PDF 내보내기(jsPDF)가 이미지 dataURL을 그대로 넘겨 JPEG/PNG만 처리되고 **WebP는 깨짐** → 브라우저가 디코드한 이미지를 흰 배경 캔버스에 그려 JPEG로 재인코딩하도록 고쳐 소스 포맷 무관(WebP 포함) 안전하게 만듦. 투명 PNG도 PDF에서 흰 배경으로 평탄화되어 들어감.

### (배치) 기존 투명 악보 240개 일괄 불투명화 — **핵심 작업**
- **발단**: 사용자가 두 이미지(육안상 동일)를 주며 차이를 물음. 운영에서 두 파일을 받아 **픽셀 알파값까지 디코드**해 확정 — 하나(file 2386)는 966×1089 RGBA로 **97.6% 투명한 원본**, 다른 하나(file 1092)는 알파 없는 **JPEG(평탄화본)**. 흰 배경에 올리면 둘이 픽셀 동일해져 구분 불가한 게 정상(차이는 다크 배경에서만 드러남).
- **드라이런(읽기 전용)**: node로 PNG를 직접 디코드(zlib inflate + 스캔라인 디필터)해 전 파일의 실제 투명 픽셀을 검사. 결과 — 총 이미지 2,460개, PNG 후보 862개 중 **실제 투명 240개**, 이미 불투명 PNG 620개, .png 이름인데 실은 JPEG 2개, 디코드 오류 0.
- **배치 실행**: node로 각 파일 다운로드 → 알파를 흰색으로 합성(`out = src·a + 255·(255−a)`) → 알파 없는 **RGB PNG(colorType 2)로 무손실 재인코딩**(직접 IHDR/IDAT/IEND 청크 + CRC 생성, filter None + deflate lv9) → `PUT /api/song-files/{id}/content`로 교체. 게스트 토큰(ROLE_USER)으로 교체 가능함을 SecurityConfig(`anyRequest().authenticated()`)로 확인. 교체 직전 재디코드해 실제 투명일 때만 처리(이미 불투명/JPEG 자동 skip), done.log 기반 재실행 가능, 401/403 시 토큰 재발급.
- **결과**: 1건 테스트(#345: 2779×3146, 투명→colorType 2 불투명 411KB 확인) 후 전량 실행 → **성공 240 / 실패 0**. 표본 5건(#345/#1355/#1897/#1372/#2566) 재검증 전부 colorType 2 불투명·해상도 보존 확인. 이제 이 악보들은 다크 뷰어·PDF·다운로드 어디서든 흰 바탕으로 표시됨. 요청대로 백업은 남기지 않음.

## 팁(알아두면 좋은 정보)

- **투명 PNG vs 평탄화본은 흰 배경 위에서 원리적으로 구분 불가**하다. 흰 배경에 합성하는 순간 투명 영역이 흰색과 동일해져 픽셀이 같아지기 때문. 채팅/미리보기가 흰 배경으로 렌더하면 아무리 자세히 봐도 차이를 못 찾는 게 정상이고, 판별하려면 **합성된 렌더가 아니라 알파 채널이 살아있는 원본 파일**을 디코드해야 한다.
- **악보(흰 바탕에 선명한 검은 선/글자)에는 JPEG가 최악**이다. 고주파 경계가 많아 JPEG가 잘 못 줄여서 오히려 PNG보다 커지고(187KB > 124KB) 선 주변에 번짐(링잉)까지 생긴다. 작게 하려면 WebP, 무손실이 필요하면 불투명 PNG를 쓸 것. "작은 JPEG 파일"을 봤다면 그건 JPEG라서가 아니라 원본 해상도가 작았던 것일 수 있으니 속단 금지.
- **"버그처럼 보이는 것"은 동일 조건 대조 테스트로 오탐을 걸러라.** dev 서버(Vite HMR)는 특정 요소만 stale 렌더로 남겨 실제 코드/프로덕션과 다른 화면을 보여줄 수 있다. 이번엔 "동일 className 신규 요소는 정상 렌더"를 확인해 없는 버그를 보고할 뻔한 걸 막았다. 프로덕션 빌드가 진짜 기준이다.
- **대량 프로덕션 덮어쓰기는 반드시 드라이런 → 1건 테스트 → 전량 순서로.** 먼저 읽기 전용으로 정확한 대상 수를 확정하고(불필요한 파일을 건드리지 않게), 1건으로 인증·인코딩·교체 전 과정을 검증한 뒤 전량 실행. 배치는 done.log 기반으로 **재실행 가능(idempotent)**하게 만들어 중단돼도 이어서 돌릴 수 있게 한다.
- **PNG는 순수 node로 디코드·인코드가 가능하다**(zlib 내장 + 스캔라인 필터/CRC 직접 구현). PIL·ImageMagick·sharp 없이도 알파 검사·흰배경 평탄화·무손실 재저장을 할 수 있어, 도구 설치 없이 서버사이드 배치가 가능했다. 단 WebP 인코딩은 순수 node로는 어려워 브라우저 canvas가 필요하다.
- **`GET /api/song-files/*/view`·`/download`만 공개**이고 `PUT .../content`는 인증 필요(관리자 아님, `anyRequest().authenticated()`). 게스트 로그인 토큰(ROLE_USER, 30일 만료)으로 콘텐츠 교체가 가능하다.
